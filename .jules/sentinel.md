## 2025-02-28 - Sentinel Journal Entry
**Vulnerability:** The Vite configuration used `define` to statically replace `process.env.GEMINI_API_KEY` and `process.env.API_KEY` with the actual API key string during the build process. Furthermore, the `geminiService.ts` file contained fallback logic looking for `process.env.GEMINI_API_KEY` and `process.env.API_KEY`. This meant that the backend API key from the local `.env` file was being injected in plaintext into the public client-side JavaScript bundle during the Vite build process.
**Learning:** Using `define` in `vite.config.ts` to map `process.env` to sensitive secrets will compile those secrets into the client-side JavaScript. This exposes secrets directly to the client browser.
**Prevention:** Avoid injecting `process.env.*` secrets into the client build. Exclusively use Vite's built-in `import.meta.env.VITE_*` mechanism for variables that are safe to be exposed to the public frontend. Remove fallback checks for `process.env` within client-side code.

## 2025-02-28 - Information Leakage & URL Validation
**Vulnerability:** The `App.tsx` file was catching errors with `any` and directly exposing raw `error.message` strings to the user via the `showAlert` UI component. This can potentially leak sensitive implementation details, stack traces, or underlying API error messages to the client. Furthermore, the `handleAnalyzeUrl` function was accepting arbitrary URLs without verifying the protocol. This could allow users to submit malicious links (like `javascript:`, `file:`, `data:`) that might be processed or forwarded unsafely, leading to XSS or SSRF risks depending on how the backend or API handles them.
**Learning:** Always provide generic, safe error messages to end users to prevent information disclosure. When accepting URLs, explicitly validate and restrict allowed protocols to known safe ones (e.g., `http:` and `https:`) to mitigate SSRF and related attack vectors. Using TypeScript's `unknown` for caught errors forces proper type-checking before accessing error properties.
**Prevention:** Implement strict `catch (error: unknown)` handling and map specific expected errors to safe messages, defaulting to a generic failure message. Validate all user-supplied URLs against an allowlist of permitted protocols before processing.
